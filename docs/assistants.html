¿En qué se diferencia de un Custom GPT?

Un Custom GPT junta tres cosas atrás de una plataforma cerrada: instrucciones, archivos subidos y un modelo fijo. Acá son las mismas tres cosas, pero como archivos de Git — auditables, versionados, sin dueño único.

Custom GPT

  • 🔒 Instrucciones ocultas en la plataforma
  • 📎 Archivos subidos, retrieval opaco
  • 🏢 Corre solo en los servidores de OpenAI
  • ❓ Nunca sabés por qué citó lo que citó

Asistente cq‑git

  • 📄 Instrucciones en un .md, revisado por PR
  • ✅ Conocimiento con cita obligatoria y confianza calculada
  • 🧩 Corre con el modelo que vos elijas
  • 🔍 Cada respuesta señala el archivo exacto que la respalda

Cómo responde una pregunta

El modelo nunca ve más de lo que el contrato del asistente permite. Si la pregunta cae fuera de su conocimiento autorizado, el modelo ni se llama — el rechazo está en el código, no depende de que el modelo "se porte bien".

✓ dentro del conocimiento permitido
Según la unidad ku_a1b2…: Stripe devuelve HTTP 200 con un error en el cuerpo cuando te limita, en vez del 429 esperado. Si tu código trata cualquier 2xx como éxito, pierde ese error silenciosamente — típicamente bajo carga. Revisá el cuerpo de la respuesta antes de confiar en el status.
📎 Fuente: Stripe API changelog — confianza 0.60

Tres formas de usarlo

Mismo contrato, mismas reglas de alcance — elegís según lo que necesites en el momento.

⌨️

Desde la terminal

Consultá el perfil, la lista de conocimiento permitido, o una unidad puntual con un solo comando. Ideal para chequear rápido o para scripts de CI.

🔌

Como herramientas MCP

Cualquier agente (Claude Code, u otro compatible) se conecta y usa el asistente como si fuera nativo — con el mismo alcance y las mismas reglas.

💬

Pregunta y respuesta real

Le hacés una pregunta y un modelo real responde, grounded solo en lo permitido. Si no hay conocimiento que la cubra, te lo dice — no inventa.

$ python scripts/assistant_run.py . soporte-stripe -- \  "¿Cómo configuro un webhook de reembolsos idempotente?" Lamentablemente, no dispongo de unidades de conocimientovinculadas que expliquen eso. Siguiendo el contrato deresponder únicamente con base en cited_only, no puedobrindar una recomendación para esa configuración.

Crear el tuyo en 4 pasos

Todo dentro de un repo cq-git-template — sin infraestructura nueva que mantener.

Escribí assistant.md

Identidad, idioma, instrucciones y política de citas — un archivo de texto, revisado por Pull Request como cualquier cambio de código.

Listá su conocimiento en knowledge.md

Solo esos IDs exactos podrá citar. Nada más — ni siquiera el resto de tu commons.

Validalo

python scripts/validate.py comprueba estructura, citas y que el índice de asistentes esté bien enlazado.

Conectalo a un modelo

CLI, MCP, o pregunta-y-respuesta end-to-end — el mismo contrato, tres formas de usarlo.

# assistants/mi-asistente/assistant.md
---
type: Assistant
id: assistant_mi_asistente
language: es
knowledge_policy: cited_only
flag_policy: warn
---
# Instructions
Respondé en español, citando siempre la fuente.
