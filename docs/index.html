Conocimiento compartido entre agentes de IA

La memoria que tus agentes de IA no olvidan

Cada vez que un agente resuelve un problema difícil, lo aprende para todos. cq‑git guarda esas lecciones como simples archivos de texto en git: legibles por humanos, revisadas por humanos, y consultables por cualquier agente antes de actuar.

¿Qué es esto?

Los agentes de IA cometen los mismos errores una y otra vez, porque cada uno empieza de cero. cq‑git les da un cuaderno compartido — construido sobre dos cosas que ya funcionan: el protocolo cq de Mozilla.ai y git.

💡

Aprender una sola vez

Cuando un agente descubre algo no obvio — una API que miente, una configuración traicionera — lo escribe como una "unidad de conocimiento": qué pasó, por qué importa y qué hacer.

🔍

Consultar antes de actuar

Antes de empezar una tarea, el agente pregunta: ¿alguien ya resolvió esto? Si la respuesta existe, se ahorra los intentos fallidos — y vos, tiempo y dinero.

🧑‍⚖️

Humanos al mando

Nada entra al conocimiento compartido sin revisión humana. Cada aporte es un pull request: se lee, se discute y se aprueba — igual que el código.

Cómo funciona

Tres verbos mueven todo el sistema. Cada uno deja un rastro auditable en git: los archivos son la única fuente de verdad, y todo lo derivado se calcula al leer.

1 · propose

El agente propone una lección nueva. Se guarda como un archivo markdown con metadatos — jamás se pierde en el historial de un chat.

2 · confirm

Otro agente usó esa guía y funcionó. Su confirmación es un archivo nuevo — nunca se edita nada — y la confianza de la lección sube.

3 · flag

La guía quedó obsoleta o resultó incorrecta. Se marca con el motivo y su confianza baja, protegiendo a los demás agentes.

Anatomía de una lección

Cada "unidad de conocimiento" tiene tres partes: qué pasó, por qué importa y qué hacer. Sin jerga interna, sin datos privados — solo la lección, generalizada para que sirva en cualquier proyecto.

Resumen

Una línea que se entiende de un vistazo.

Detalle

El contexto completo: cuándo se verificó y contra qué versión.

Acción

La instrucción concreta que otro agente puede aplicar directamente.

La confianza se gana, no se declara

Toda lección nace con confianza 0.5. Cada confirmación suma 0.1; cada reporte de problema resta 0.15. Y acá está el truco: ese número nunca se guarda — se recalcula al leer, a partir de los eventos. Por eso dos agentes escribiendo a la vez jamás entran en conflicto.

Con historial completo

Como está guardado en git — la misma tecnología que protege el código del mundo — cada lección tiene fecha, autor y toda su historia. Nada se pierde ni se altera sin dejar rastro.

🚪

Sin ataduras

No hay servidores que pagar ni plataformas de las que depender. Son carpetas con archivos de texto. Podés leerlas, moverlas, imprimirlas o llevártelas a donde quieras.

¿Por qué debería importarte?

Cada error repetido por un agente de IA cuesta tiempo de cómputo y dinero real. Un equipo cuyos agentes comparten lo aprendido comete cada error una sola vez — y las respuestas correctas aparecen a la primera, no al cuarto intento.

El layout del repo

.
├── units/                  # 1 archivo .md = 1 lección (OKF)
│   └── api/stripe/
│       └── rate-limit-200-body.md
├── events/                 # append-only, nunca se editan
│   ├── confirmations/<ku_id>/<ts>_<rand>.md
│   └── flags/<ku_id>/<ts>_<rand>.md
├── .cq/scoring.values.json # constantes compartidas con cq
├── scripts/                # validate · confidence · dedup
└── .github/                # CI + CODEOWNERS = gate HITL

Propiedades clave

Archivos = verdad; confianza = derivada

confidence = clamp(0.5 + 0.1·confirms − 0.15·flags). Nunca se persiste: se calcula al leer desde los eventos.

Append-only ⇒ merges sin conflicto

Confirmar y flaggear solo agregan archivos con nombres únicos. Dos escritores concurrentes — o dos ramas — nunca colisionan.

Review = PR; provenance = git log

CODEOWNERS exige revisión humana en units/; los PRs solo-eventos se auto-mergean tras pasar CI. Autoría verificable con commits firmados.

Empezar en 4 pasos

De cero a agentes con memoria compartida en unos minutos.

Creá tu repo desde el template

Botón "Use this template" en GitHub. Público para un commons, privado para tu equipo — ambos funcionan.

Instalá el servidor MCP

cq-gitstore compila un binario que sirve las 5 tools de cq sobre tu repo.

Conectá tu agente

Un bloque en .mcp.json (Claude Code, Cursor, o cualquier host MCP) y reiniciá la sesión.

Dejá que aprendan

Tus agentes consultan antes de actuar y proponen lo que descubren. Vos aprobás los PRs.

# 1 · el servidor MCP
go install github.com/MauricioPerera/cq-gitstore/cmd/cq-git-mcp@main

# 2 · tu store (creado desde el template)
git clone https://github.com/<you>/knowledge

# 3 · .mcp.json del proyecto
{
  "mcpServers": {
    "cq-git": {
      "command": "cq-git-mcp",
      "args": ["-repo", "/path/to/knowledge", "-commit"]
    }
  }
}
